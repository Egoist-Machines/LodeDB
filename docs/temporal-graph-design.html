LodeDB · design proposal

lodedb-graph: a bi-temporal knowledge graph

A native Rust graph layer over lodedb-core that remembers when things were true. The temporal successor to lodedb.graph.KnowledgeGraph — same architecture, plus event-time + transaction-time validity, fact invalidation instead of deletion, and “as-of T” traversal. Built once in Rust, bound to Python and Swift, so it runs on a server and on a phone.

01Why this exists

LodeDB is an exact, embedded vector index — not a graph engine. Its own docs/graph.md already makes the case that graph-backed memory (Zep/Graphiti, cognee) doesn’t need LodeDB to be a graph; it needs a fast local semantic-retrieval layer paired with a store built for traversal. LodeDB ships exactly that pairing today as the Python lodedb.graph.KnowledgeGraph: an authoritative SQLite topology store, with LodeDB as a rebuildable semantic index over it.

Two things are missing, and both matter for any system that models a life, a project history, or an evolving world:

  • No sense of time. KnowledgeGraph edges are timeless. A fact is either present or absent; there is no record of when it became true, when it stopped being true, or when the system learned it. You cannot ask “who did Alice work for in 2019?” once she changes jobs — the old edge is simply overwritten or deleted.
  • It is Python-only. The topology store is sqlite3 in Python. On-device consumers (an iOS app) have the Swift LodeDB vector store but no graph layer at all — so they reinvent a weak “graph” by stuffing src/dst strings into flat document metadata and calling listDocuments(filter:{src:x}) a traversal. That has no typed edges, no k-hop, and no time.

This proposal is a native Rust crate, lodedb-graph, that keeps the proven two-store architecture, adds full bi-temporality, and — being native — binds to both Python and Swift so the same engine runs on a server and on a phone.

02First principles

Principle 1 — one source of truth

The topology store (nodes, typed edges, validity intervals) is authoritative. LodeDB is a derived, rebuildable index over it. If an index write is lost, the graph is still correct and reindex() restores it — so there is no cross-store atomicity problem to get wrong. This is the invariant KnowledgeGraph already holds; we keep it.

Principle 2 — record time, never destroy it

Facts are invalidated, not deleted. A new, contradicting fact closes the old one’s validity interval rather than erasing it. The graph therefore accumulates history for free, and “the current view” is just a query — the special case of “as of now”.

Principle 3 — two clocks, not one

Every fact carries event time (when it was true in the world) and transaction time (when the system learned or retracted it). These are independent: you can learn today that someone moved cities last year. Bitemporality is what lets the graph answer both “what was true then?” and “what did we believe then?”.

Principle 4 — the engine is generic

Entity type and edge relation are caller-defined strings. The crate ships no domain vocabulary. Applications (an agent memory, a personal “life” graph, a CRM) define their own types on top. The worked example in §10 is one such application, not part of the engine.

03Architecture

Two stores behind one handle, exactly mirroring KnowledgeGraph’s split — the only structural change is that the topology store gains temporal columns and the whole thing is native Rust.

Topology truth store

An embedded SQL database (rusqlite, bundled). Holds the complete, authoritative graph and every validity interval. All traversal (neighbors, k_hop), enumeration, and temporal resolution runs here, in SQL, where range predicates over time are natural. Crash-durable and transactional per write.

Semantic index

An in-process lodedb-core engine, driven exactly as KnowledgeGraph drives LodeDB. Indexes each entity’s label and each fact’s fact text for hybrid (vector + BM25) entry-point search. Fully derived: reindex() rebuilds it from the truth store, dropping orphans.

04The bi-temporal model

Three record kinds, following Graphiti’s ingestion spine — episodes → entities → facts.

Episode

A raw observation the graph was built from: a note, a chat turn, a calendar event, a connector payload. Episodes carry provenance and an occurred_at (Graphiti’s reference_time), and a source — typically message, text, or json, matching Graphiti’s EpisodeType. Entities and facts reference the episodes they were extracted from, so every derived assertion is traceable to a source. Episodes are the natural home for an ingestion / refine-enrich loop.

Entity (node)

A thing in the world: an id, a caller-defined type, a human label (the text that gets embedded), free-form JSON properties, a created_at, and — for entities that begin and end (a project, an event, a job) — an optional validity interval.

Fact (edge)

A typed, directed, labeled relationship (src) —[relation]→ (dst) with JSON properties, a natural-language fact string (indexed for semantic edge search), and four timestamps:

TimestampClockMeaningNull means
valid_ateventwhen the fact became true in the worldunknown / always
invalid_ateventwhen it stopped being true in the worldstill true
created_atsystemwhen the system recorded the fact—
expired_atsystemwhen the system retracted / superseded itstill believed

The two clocks are what make the model bitemporal. Event time answers “what was true then”; system time answers “what did we believe then”. A correction that says “actually, she left in March, not June” closes the wrong fact on the system clock (expired_at) and writes a new one with the corrected event clock.

Fidelity to Graphiti’s model

The field names and semantics track Graphiti’s graphiti_core EntityEdge exactly, so this is a faithful embedded reimplementation of its temporal core, not a loose analogy:

Graphiti (graphiti_core)This designMeaning
EntityEdge.valid_atfacts.valid_atevent time — fact became true
EntityEdge.invalid_atfacts.invalid_atevent time — fact stopped being true
EntityEdge.created_atfacts.created_attransaction time — recorded
EntityEdge.expired_atfacts.expired_attransaction time — superseded / retracted
EntityEdge.reference_timefacts.reference_timeevent-time anchor of the source episode
EpisodicEdge (MENTIONS)episode_mentionsepisode → entity provenance
EntityEdge.episodesfact_episodesfact → source episodes

Invalidation is Graphiti’s exact rule: on a contradicting fact, close the prior edge by setting its expired_at (transaction axis) to now and its invalid_at (event axis) to the new fact’s valid_at — never a delete. A “live” fact is expired_at IS NULL.

What this is NOT — the boundary with Graphiti

Graphiti is an LLM-in-the-loop pipeline. Its add_episode(text) requires an llm_client + embedder + cross_encoder and runs entity extraction, embedding-based entity resolution (dedup / merge against existing nodes), LLM temporal extraction (it reads valid_at/invalid_at out of the prose, anchored to the episode’s reference_time), and LLM contradiction detection to decide what to invalidate.

lodedb-graph is the storage + temporal-query half of that. It stores episodes, entities, and facts, resolves as-of / history queries, and invalidates on demand — but the caller supplies the extracted timestamps and decides what contradicts what. Its add_fact(valid_at=…, invalidates=…) is the direct analogue of Graphiti’s LLM-free add_triplet(); a full Graphiti-style ingest is a thin LLM layer on top. This mirrors how LodeDB keeps embedding pluggable rather than built in: storage infra, bring-your-own-model.

Deliberately out of scope (Graphiti has them; the engine won’t, at least in v1)

Communities — Graphiti’s third subgraph tier (Leiden clustering + incremental label-propagation, CommunityNode/HAS_MEMBER) is an optional summarization layer; deferred, addable later as a communities table. Advanced rerankers — Graphiti ships RRF, MMR, center-node/graph-distance, episode-mention, and cross-encoder rerank; we ship hybrid vector+BM25+RRF seeds plus exact k-hop, with a center-node rerank an easy follow-on. Typed edge maps (edge_type_map, constraining which relations may join which entity-type pairs) are extraction guidance, so they belong with the caller’s LLM layer, not the store. Backends — Graphiti targets server graph DBs (Neo4j / FalkorDB / Kuzu / Neptune); we target embedded SQLite + LodeDB, deliberately, so it runs on a phone.

05Topology schema

The authoritative store, as SQL DDL. It extends KnowledgeGraph’s _store.py schema (nodes / edges / node_vectors, indexed by type / src / dst / relation) with an episodes table and temporal columns. Timestamps are integer epoch-milliseconds; NULL encodes open intervals.

CREATE TABLE episodes (
  id          TEXT PRIMARY KEY,
  source      TEXT NOT NULL,          -- caller-defined: note | chat | calendar | …
  body        TEXT NOT NULL DEFAULT '',
  occurred_at INTEGER NOT NULL,       -- event time of the observation (epoch ms)
  created_at  INTEGER NOT NULL,       -- system time it was ingested
  properties  TEXT NOT NULL DEFAULT '{}'
);

CREATE TABLE entities (
  id          TEXT PRIMARY KEY,
  type        TEXT NOT NULL DEFAULT '',   -- caller-defined vocabulary
  label       TEXT NOT NULL DEFAULT '',   -- embedded for semantic search
  properties  TEXT NOT NULL DEFAULT '{}',
  valid_at    INTEGER,                  -- optional entity validity (a project/event span)
  invalid_at  INTEGER,
  created_at  INTEGER NOT NULL,
  expired_at  INTEGER
);

CREATE TABLE facts (
  id          TEXT PRIMARY KEY,
  src         TEXT NOT NULL,
  relation    TEXT NOT NULL,          -- the labeled edge type
  dst         TEXT NOT NULL,
  fact        TEXT NOT NULL DEFAULT '',   -- natural-language, indexed for edge search
  properties  TEXT NOT NULL DEFAULT '{}',
  valid_at       INTEGER,                  -- event time: true-from / true-until
  invalid_at     INTEGER,
  created_at     INTEGER NOT NULL,       -- system time: recorded / retracted
  expired_at     INTEGER,
  reference_time INTEGER                   -- event-time anchor of the source episode (Graphiti parity)
);

CREATE TABLE fact_episodes (   -- provenance: which episodes asserted a fact (EntityEdge.episodes)
  fact_id     TEXT NOT NULL REFERENCES facts(id) ON DELETE CASCADE,
  episode_id  TEXT NOT NULL REFERENCES episodes(id) ON DELETE CASCADE,
  PRIMARY KEY (fact_id, episode_id)
);

CREATE TABLE episode_mentions (   -- provenance: which entities an episode mentions (Graphiti MENTIONS)
  episode_id  TEXT NOT NULL REFERENCES episodes(id) ON DELETE CASCADE,
  entity_id   TEXT NOT NULL REFERENCES entities(id) ON DELETE CASCADE,
  PRIMARY KEY (episode_id, entity_id)
);

-- traversal + temporal-resolution indices
CREATE INDEX idx_facts_src ON facts(src, relation);
CREATE INDEX idx_facts_dst ON facts(dst, relation);
CREATE INDEX idx_facts_valid ON facts(valid_at, invalid_at);
CREATE INDEX idx_facts_live ON facts(expired_at);   -- current-belief filter
CREATE INDEX idx_mentions_entity ON episode_mentions(entity_id);
CREATE INDEX idx_entities_type ON entities(type);
A note on the truth-store backend

rusqlite (bundled SQLite) is the recommendation: it mirrors the existing Python KnowledgeGraph schema, keeps the two implementations conceptually aligned, and makes as-of range predicates trivial. Our crate keeps unsafe_code = "forbid" (rusqlite’s unsafe is in-dependency). The one cost is a bundled-C dependency in an otherwise CBLAS-only Rust build; if that is unwelcome upstream, a pure-Rust redb backend with hand-built secondary indices is the alternative. See §12.

06Temporal semantics

Every read has an implicit or explicit temporal frame. Three canonical frames, each a predicate over the four columns:

FramePredicateAnswers
Current (default) expired_at IS NULL AND invalid_at IS NULL “What is true and believed right now?”
As-of T (event) valid_at <= T AND (invalid_at IS NULL OR invalid_at > T) “What was true in the world at time T?”
History (no temporal filter) “Everything ever asserted about this entity.”

The as-of frame threads through traversal too: neighbors, k_hop, and search_subgraph all accept an optional as_of, so a two-hop expansion “as of 2019” only crosses facts valid in 2019. A system-time frame (“as we believed on date D”, adding created_at <= D AND (expired_at IS NULL OR expired_at > D)) is available for audit/debugging but off the common path.

Invalidation, worked

add_fact with invalidates = [prior_id] (or an automatic “same src+relation, single-valued” policy) performs, in one transaction:

-- 1. close the superseded fact on BOTH clocks
UPDATE facts SET invalid_at = :new_valid_at,   -- world: stopped being true here
                 expired_at = :now             -- belief: we no longer assert it as current
 WHERE id = :prior_id AND expired_at IS NULL;

-- 2. insert the new fact, open on both clocks
INSERT INTO facts (id, src, relation, dst, fact, valid_at, created_at)
       VALUES (:id, :src, :relation, :dst, :fact, :new_valid_at, :now);

Nothing is deleted; history() returns the full chain, and reindex() can drop the superseded fact from the index (so current-view search stays clean) while the truth store keeps it.

07Semantic index & the as-of encoding

The index half is KnowledgeGraph’s, unchanged in spirit: each entity is indexed by label (or a caller-supplied vector), each fact optionally by its fact text, under an id-prefix (n:/e:) with kind/type metadata. One addition makes temporal search possible at the vector layer.

Timestamps as sortable strings

LodeDB metadata is string→string, and its filter grammar compares numerically when both sides parse as numbers. So each indexed record carries its validity as zero-padded epoch-millis strings (valid_at, invalid_at, expired_at). An as-of query then becomes a native MetadataFilter the engine applies before top-k:

{ "$and": [
  { "kind": "edge" },
  { "valid_at":   { "$lte": "00001577836800000" } },
  { "invalid_at": { "$gt":  "00001577836800000" } }
]}

This is the same predicate Graphiti applies in its own SearchFilters, which expose all four temporal fields (valid_at/invalid_at/created_at/expired_at) as date comparisons (>=, <=, …) before reranking — here it runs one layer down, inside the vector engine’s filter.

Authoritative integer timestamps live in the truth store; the string mirror exists only so semantic entry-point search can be time-scoped without a post-filter. search_subgraph uses this for the seed step, then does the exact as-of traversal in SQL.

08Public API (Rust)

Mirrors KnowledgeGraph’s surface and adds the temporal verbs. Text-in (the engine embeds label/fact) and vector-in (bring-your-own embeddings) are both supported, as in the Python layer.

MethodPurpose
open(path, IndexConfig)Open/create the truth store + index at a directory.
add_episode(source, body, occurred_at, props)Record a raw observation; returns its id.
upsert_entity(id?, type, label, props, valid?)Create/replace an entity and (re)index it.
add_fact(src, relation, dst, fact, valid_at, episodes, invalidates?)Assert a fact; optionally close superseded facts.
invalidate_fact(id, at)Close a fact’s event + system validity without a replacement.
entities(type?, as_of?)Complete-set enumeration by type in a temporal frame.
neighbors(id, dir, relation?, as_of?)Incident facts (out/in/both), time-scoped.
k_hop(seeds, k, dir, as_of?)Deterministic BFS neighbourhood in a temporal frame.
semantic_entities(query|embedding, k, type?, filter?, as_of?)Hybrid vector+BM25 entry points, optionally as-of.
semantic_facts(query|embedding, k, relation?, as_of?)Hybrid search over fact text, returns facts — Graphiti’s default search() shape.
search_subgraph(query, k, hops, as_of?)Semantic seeds + k-hop expansion (the headline query).
history(id)Every fact/version touching an entity, all frames.
reindex() · stats() · persist()Rebuild index from truth · counts · checkpoint.
 
// as-of traversal, end to end
let g = TemporalGraph::open("./life", IndexConfig::minilm())?;
let ep = g.add_episode("note", "Alice joined Globex", ms("2021-04-01"), json!({}))?;
g.add_fact("alice", "works_at", "globex", "Alice works at Globex",
           ms("2021-04-01"), &[ep], Some(&["alice:works_at:acme"]))?;

let now  = g.neighbors("alice", Dir::Out, Some("works_at"), None)?;         // → Globex
let then = g.neighbors("alice", Dir::Out, Some("works_at"), Some(ms("2019-01-01")))?; // → Acme

09Bindings — one engine, three surfaces

Because the engine is native, it reaches every LodeDB surface through mechanisms the repo already uses.

🦀 Rust

The crate itself: crates/lodedb-graph, a lodedb-core path-dependency, [lints] workspace = true. The template is lodedb-cloud-core — a pure-Rust crate on lodedb-core.

🐍 Python

A pyo3 submodule in turbovec-python, mirroring cloud.rs’s register(): lodedb.graph gains a TemporalKnowledgeGraph class alongside the existing KnowledgeGraph. No C ABI involved.

🍎 Swift / on-device

A LodeGraph opaque handle + lodedb_graph_*_json functions in lodedb-ffi (JSON-in/JSON-out, u32 status), a hand-updated header, an ABI bump, and a LodeGraph Swift class beside NativeEngine. This is what puts a real graph on the phone.

Relationship to KnowledgeGraph

The Python KnowledgeGraph stays for back-compat. TemporalKnowledgeGraph is the superset: temporal, native, and available on-device. No change to lodedb-core storage or the public SDK.

10Consumer example — a “LIFE” graph

To show the engine in use (and to motivate it): a personal timeline app models a user’s world with seven caller-defined entity types — Event · Person · Place · Project · Plant · Animal · Thing — and typed, labeled facts between them (owns, lives_at, attended, works_on, gifted, …).

Everything temporal falls out of the model rather than being special-cased:

  • An Event is an entity with a validity span; “what happened in 2023?” is entities("Event", as_of: 2023-…) — or a range scan.
  • A Person’s job, home, or relationship history is the fact chain on works_at / lives_at; changing jobs invalidates the old fact instead of erasing it.
  • A Plant or Animal acquired and later lost is a owns fact with a closed interval — the graph still knows you had it.
  • “Who did I know when I lived in Lisbon?” is a two-hop search_subgraph with as_of set to that period.
Adoption shape (application side, out of scope for the crate)

A consumer adopts the graph half only: entities and facts move into TemporalKnowledgeGraph’s truth store, while entity/fact text is still indexed into the app’s existing LodeDB store so plain-document search stays unified. Domain choices like retiring an old org type into Thing, labeling edges, and deleting facts on unlink are the application’s vocabulary decisions — the engine only provides the primitives.

Verification

  • Rust — cargo test -p lodedb-graph: CRUD, typed traversal, k-hop, and the temporal invariants (invalidation supersedes; as-of-T returns the fact valid at T; history is complete; reindex rebuilds from truth). Clean under workspace clippy.
  • Python — pytest parity with the Rust behaviours plus the existing bulk-load / hybrid / reindex-orphan-drop patterns.
  • Swift — swift test: a LodeGraphTests mirroring the behaviours on device, including an as-of query and an invalidation round-trip.
  • End-to-end — the metrics-only graph_memory harness extended with ingest → contradict → as-of.
